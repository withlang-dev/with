# Package Management — Spec & Implementation Notes

## Overview

With has two dependency sources:

- **With packages** — `with get json`, `with get http`
- **C packages** — `with get c.glib`, `with get c.sqlite3`

The `c.` prefix routes through Conan Center (conan.io/center).
With packages come from the With package registry (future).
Both are managed through the same CLI and `with.toml`.

## CLI Commands

### `with init`

Creates a new project:

```
mkdir myapp && cd myapp
with init
```

Produces:

```
myapp/
├── with.toml
└── src/
    └── main.w
```

`with.toml`:
```toml
[project]
name = "myapp"
version = "0.1.0"
```

`src/main.w`:
```
fn main:
    print("Hello, World!")
```

Options:
```
with init              # interactive — asks for name, creates src/main.w
with init --lib        # library project — creates src/lib.w, no main
with init --name foo   # non-interactive, uses "foo" as project name
```

### `with get`

Adds a dependency:

```
with get c.glib             # latest stable from Conan Center
with get c.glib@2.78        # specific version
with get c.sqlite3           # sqlite3 from Conan
with get c.openssl           # openssl from Conan
```

This command:
1. Resolves the package from Conan Center
2. Downloads headers, prebuilt libraries, and transitive deps
3. Caches everything in `.with/deps/c/<name>/<version>/`
4. Updates `with.toml` with the dependency entry

After `with get c.glib`:

`with.toml`:
```toml
[project]
name = "myapp"
version = "0.1.0"

[deps]
c.glib = "2.78"
```

`.with/deps/`:
```
.with/deps/c/glib/2.78/
├── include/
│   └── glib-2.0/
│       ├── glib.h
│       ├── glib/ghash.h
│       └── ...
├── lib/
│   ├── libglib-2.0.a
│   ├── libpcre2-8.a      (transitive dep)
│   └── libffi.a          (transitive dep)
└── metadata.json
```

`metadata.json` (generated by Conan, consumed by With):
```json
{
  "name": "glib",
  "version": "2.78",
  "include_paths": ["include/glib-2.0", "include"],
  "lib_paths": ["lib"],
  "libs": ["glib-2.0", "pcre2-8", "ffi"],
  "defines": [],
  "transitive_deps": ["pcre2", "libffi"]
}
```

### `with remove`

```
with remove c.glib
```

Removes the dependency from `with.toml` and optionally cleans
`.with/deps/c/glib/`.

### `with update`

```
with update             # update all deps to latest compatible
with update c.glib      # update specific dep
```

## Build Integration

### Include Path Resolution

When `with build` encounters `use c_import("<glib.h>")`:

1. Check standard system include paths (default behavior)
2. Check `.with/deps/c/*/include/` for all installed C deps
3. The include paths from `metadata.json` are passed to
   `cc -E` (the C preprocessor) via `-I` flags

The compiler reads `with.toml`, finds all `c.*` deps, reads
their `metadata.json`, and constructs the include path list
automatically. The user never writes `-I` flags.

### Link Resolution

When `with build` links the final binary:

1. Read `with.toml` for all `c.*` deps
2. For each dep, read `metadata.json` for `libs` and `lib_paths`
3. Pass `-L` and `-l` flags to the linker automatically

```
# What the compiler generates internally:
clang -o myapp myapp.o \
  -L.with/deps/c/glib/2.78/lib \
  -lglib-2.0 -lpcre2-8 -lffi
```

The user never sees link flags.

### Header-to-Package Mapping

When the compiler processes `use c_import("<glib.h>")`, it needs
to find which package provides `glib.h`. The resolution order:

1. Search each installed C dep's include paths for the header
2. If found in exactly one package, use that package's metadata
   for link flags
3. If found in multiple packages, use the first match (or error
   with "ambiguous header" if the packages conflict)
4. If not found in any dep, fall back to system include paths

This means the user writes `use c_import("<glib.h>")` — not
`use c_import("<glib.h>", link: "glib-2.0")`. The `link:` 
parameter is still available for headers outside the package
system (system-installed libraries, custom builds).

### Explicit Link Override

If the auto-resolution picks the wrong library, or if the user
wants to link additional libraries:

```
use c_import("<glib.h>", link: "glib-2.0", "gio-2.0")
```

The explicit `link:` overrides the auto-resolved metadata.

## Conan Integration

### Why Conan

- 1,500+ C/C++ packages in Conan Center
- Prebuilt binaries for common platforms
- Handles transitive dependencies
- MIT licensed, industry-standard
- Active: new packages added weekly

### Implementation

`with get c.X` shells out to the Conan CLI:

```
conan install --requires=X/latest \
  --output-folder=.with/deps/c/X/VERSION \
  --build=missing \
  --generator=JSONGenerator
```

The `JSONGenerator` produces the metadata file that With reads.
If Conan isn't installed, `with get c.X` prints:

```
error: Conan is required for C package management.
Install it with: pip install conan
```

### Conan-Free Fallback

For users who don't want Conan, manual C deps work:

```toml
# with.toml
[deps.c.glib]
include = "/usr/include/glib-2.0"
lib = "/usr/lib"
link = ["glib-2.0"]
```

This is the escape hatch — no Conan, no download, just point
at your system installation. `c_import` uses these paths.

### Platform Handling

Conan handles cross-platform binaries. `with get c.glib` on
macOS gets macOS binaries. On Linux, Linux binaries. The With
compiler doesn't need to know — Conan's profile system handles
it.

For cross-compilation:

```
with get c.glib --target=aarch64-linux
```

Passes the target profile to Conan.

## .with Directory

```
.with/
├── deps/
│   └── c/
│       ├── glib/2.78/
│       │   ├── include/
│       │   ├── lib/
│       │   └── metadata.json
│       └── sqlite3/3.45/
│           ├── include/
│           ├── lib/
│           └── metadata.json
├── cache/
│   └── c_import/           # existing c_import cache
└── lock.json               # dependency lock file
```

`.with/` should be in `.gitignore`. `with.toml` and `lock.json`
are committed. `with get` (with no arguments) restores deps
from the lock file — like `npm install` with a `package-lock.json`.

## with.toml Full Schema

```toml
[project]
name = "myapp"
version = "0.1.0"
authors = ["Eric"]
description = "My application"

[deps]
# With packages (future)
# json = "1.0"
# http = "0.5"

# C packages via Conan
c.glib = "2.78"
c.sqlite3 = "3.45"
c.openssl = "3.2"

# C packages with manual paths (no Conan)
[deps.c.custom_lib]
include = "/opt/custom/include"
lib = "/opt/custom/lib"
link = ["custom"]

[build]
target = "native"           # or "aarch64-linux", "x86_64-windows"
optimize = "release"        # "debug", "release", "small"

[c_import]
cc = "cc"                   # C compiler for preprocessing
extra_includes = []         # additional -I paths (beyond deps)
extra_defines = {}          # additional -D flags
```

## Implementation Order

```
Phase 1: with init (~50 lines)
  - Create with.toml and src/main.w
  - Read with.toml in compiler for project name

Phase 2: with.toml dep reading (~80 lines)
  - Parse [deps] section in compiler
  - Construct include paths and link flags from metadata.json
  - Pass to c_import and linker

Phase 3: with get c.X (~120 lines)
  - Shell out to conan install
  - Parse Conan JSON output into metadata.json
  - Update with.toml

Phase 4: Header-to-package auto-resolution (~60 lines)
  - Search dep include dirs for headers
  - Auto-link the right libraries without link: parameter

Phase 5: with remove, with update (~40 lines each)
  - Remove/update entries in with.toml
  - Clean/refresh .with/deps

Phase 6: lock.json (~50 lines)
  - Record exact versions after resolution
  - with get (no args) restores from lock
```

Total: ~400 lines for the full package management flow.
The compiler changes (Phase 2 and 4) are ~140 lines —
everything else is CLI tooling.
